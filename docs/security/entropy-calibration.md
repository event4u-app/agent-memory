# Entropy-Heuristic Calibration

The `HIGH_ENTROPY_DETECTED` heuristic is the residual detector that runs
**after** the named-pattern catalog (`src/security/secret-patterns.ts`).
It flags any quoted ≥ 20-char string whose Shannon entropy exceeds a
configurable threshold. The threshold is not guessed — it is calibrated
against a fixture corpus so operators have evidence for where they set
the dial.

## Corpus

- `tests/fixtures/entropy-corpus/non-secrets.txt` — legitimate strings
  that would trigger false positives if the threshold is too low: Git
  SHAs, UUIDs, hashes, base64 of small binaries, ARNs, long paths,
  bundle filenames, and natural language fragments.
- `tests/fixtures/entropy-corpus/secrets.txt` — syntactically
  random-looking tokens that carry no catalog prefix (catalog-shaped
  tokens live in `tests/e2e/canaries.ts` instead). The
  `agent_memory_canary` marker is embedded where the length budget
  allows, so any leak into production logs is trivially searchable.

Strings that also match a named catalog pattern are excluded from the
residual measurement — the eval only reports on lines the catalog
*cannot* catch on its own.

## Methodology

For each candidate threshold, `scripts/eval-entropy-threshold.ts` runs
every corpus line through the production guard (`scanForSecrets`) with
only the entropy threshold overridden. A line is a **true positive** if
flagged in `secrets.txt` and a **false positive** if flagged in
`non-secrets.txt`. Precision, recall, and F1 are computed on the
residual set. The script picks the F1-maximizing threshold and breaks
ties in favor of fewer false positives, since every FP pushes agents to
work around the guard.

## Results

Regenerate with `npm run entropy:eval`. CI runs the same command and
fails on a `git diff` against this file, so the reasoning stays in sync
with the catalog and corpus.

### Why the default landed at `4.5`

- `3.5` trips on every Git SHA-40 (entropy ≈ 4.0 across the 16-char hex
  alphabet) and most UUIDs. 66 false positives out of 108 is unusable;
  agents would disable the guard.
- `4.0` (the previous default) still misfires on SHA-256 hashes and long
  base64 filenames — 15 false positives, half of them legitimate
  lockfile content.
- `4.5` keeps 96 % precision with 74 % recall on *residual* secrets
  (catalog-shaped tokens are caught upstream and excluded from this
  measurement). False positives drop to 3, all on long mixed-case +
  digit fragments that a reviewer would flag anyway.
- `5.0` sacrifices half the recall for one extra FP removal.

F1 is the deciding metric with a tiebreaker on FPs — operators pay for
every false positive in agent friction. The catalog is the primary
defense; this heuristic is a backstop.

## Configuration

The chosen threshold is the default for `MEMORY_ENTROPY_THRESHOLD`. The
minimum length is `MEMORY_ENTROPY_MIN_LENGTH` (default 20). Operators
can tighten either via environment variables without a rebuild — see
`src/config.ts` and the env list in the root `README.md`.

Tightening the threshold lowers recall on never-seen token shapes;
loosening it raises false positives on Git SHAs and hashes. The
corpus + eval together are the contract that lets the team move this
knob with evidence.
